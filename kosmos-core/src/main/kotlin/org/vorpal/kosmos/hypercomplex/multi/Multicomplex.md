# Plans for Multicomplex Implementation

- Bicomplex numbers are not a dead end
- They are the n = 2 case of multicomplex numbers
- The clean extension is to represent the general object in the primitive idempotent basis, where multiplication becomes componentwise
- My hesitation would be about design clarity, not possibility

### The core reason it works

Your bicomplex implementation is already exploiting the crucial fact: bicomplex numbers are generated by two commuting imaginary units $i$ and $j$, with:
$$i^2 = -1$$
$$j^2 = -1$$
$$ij = ji$$

That is not Cayley-Dickson territory. It is a commutative algebra built by adjoining commuting square roots of $-1$.

The multicomplex numbers do the same thing with more commuting imaginary units:

$$\mathbb{M}_n = \mathbb{R}[i_1,\dots,i_n] / (i_1^2+1,\dots,i_n^2+1,\ i_a i_b - i_b i_a)$$

This has real dimension $2^n$.

So bicomplex numbers are just:

$$\mathbb{M}_2$$

and ordinary complex numbers are:

$$\mathbb{M}_1$$

### Why your bicomplex representation generalizes

The secret engine in your implementation is not really the standard basis:

$$\{1, i, j, ij\}.$$

It is this:
- You found an idempotent decomposition.
- In that basis, multiplication is componentwise over `Complex`.

That is the gold vein.

For bicomplex numbers, you effectively have:

$$\mathbb{M}_2 \cong \mathbb{C} \oplus \mathbb{C}$$

with primitive orthogonal idempotents hiding behind the curtain.

For tricocomplex / multicomplex numbers, this continues:

$$\mathbb{M}_n \cong \mathbb{C}^{\,2^{n-1}}$$

as a ℂ-algebra.

That means an $n$-complex number can be represented as a tuple of $2^{n-1}$ ordinary complex numbers, with:
- addition componentwise
- multiplication componentwise
- reciprocal exists iff every component is nonzero

That is extremely nice.

So from a purely algebraic standpoint, your bicomplex code is already waving a flag that says:

“Build me recursively or flatten me into idempotent coordinates.”

### What I would object to

Not the extension itself. I would object to extending it in the same ad hoc style as `Bicomplex`.

Your current Bicomplex is elegant because the size is small enough that you can afford hand-crafted:
- $a, b, c, d$
- $z1, z2$
- $\alpha, \beta$
- three named conjugations
- special-case norms
- special pretty-printer labels

That stops scaling very quickly.

By the time you hit multicomplex numbers:

1. Basis explosion

The real dimension is $2^n$.

So:
- bicomplex: 4
- tricomplex: 8
- quadricomplex: 16
- pentacomplex: 32

A bespoke data class with named coefficients becomes unmanageable very quickly.

2. Conjugation explosion

For $n$ commuting imaginary units, each unit may or may not be conjugated.

That gives $2^n - 1$ nontrivial involutions.

So:
- bicomplex has 3
- tricomplex has 7
- quadricomplex has 15
- 
At that point, “the conjugation” is gone. There is a whole family of them.

3. Convention explosion

The mathematics is fine, but the implementation can go bad if conventions are not frozen early.

You must choose once and for all:
- standard basis ordering
- primitive idempotent ordering
- recursive split convention
- butterfly branch order
- whether the level-2 generic conversion matches the existing Bicomplex($\alpha$, $\beta$) exactly

4. Norm ambiguity

In bicomplex land, you can still talk coherently about `norm1`, `norm2`, `norm3`.

In multicomplex land, there is no single obvious distinguished norm. There are many norm-like constructions attached
to different involutions.

5. Zero divisors everywhere

In the idempotent model
$$\mathbb{C}^{2^{n-1}}$$
an element is invertible iff every coordinate is nonzero.

So zero divisors are not an edge case. They are built into the geometry of the algebra.

6. Standard-basis conversion complexity

For bicomplex numbers, converting between standard and idempotent coordinates is still human-sized.

For general multicomplex numbers, the transform becomes a recursive complex-twisted Walsh-Hadamard pattern.
It is still doable, but it is not something to hand-code separately for each level.

The implementation targets should be:
- `standardRealsToIdempotent(level, coeffs)`
- `idempotentToStandardReals(z)`

These must be tested as mutual inverses before they are trusted by printables, conjugations, or norm-like constructions.

This conversion machinery is not part of the absolute minimal core type, but it is part of the first serious milestone. It should be implemented early, because everything else interesting depends on getting these conventions right.

### So what should the general design be?

I think there are two sensible designs.

### Option A: recursive commuting-complex extension

This mirrors the definition:

$$\mathbb{M}_{n+1} = \mathbb{M}_n[j] / (j^2 + 1)$$

with $j$ commuting with everything in $\mathbb{M}_n$.

So a value looks like:

```kotlin
data class MultiComplex<A>(
    val x: A,
    val y: A
)
```

interpreted as:

$$x + yj \quad\text{with}\quad j^2 = -1$$

- If A = Complex, you get bicomplex.
- If A = Bicomplex, you get tricomplex.
- If A = Tricomplex, you get quadricomplex.

Multiplication

Exactly the ordinary complex formula, but over a commutative base ring:

$$(x + yj)(u + vj) = (xu - yv) + (xv + yu)j$$

This is perfectly valid provided the base ring is commutative.

Why this is good
- recursive
- elegant
- mirrors the mathematics
- works with your existing style

Why this is not enough
- reciprocal is harder to express directly
- zero divisors become less transparent
- canonical printing / basis naming gets messy
- idempotent decomposition is not explicit

This is the “beautiful theory-first” design.

### Option B: primitive idempotent representation over Complex

This is the design I’d favor for actual use.

Represent an $n$-complex number by a list of $2^{n-1}$ complex coordinates in primitive idempotent form:

```kotlin
data class MultiComplex private constructor(
    val components: List<Complex>
)
```

with invariant:
- `components.size` is a power of two
- `level` is inferred from components.size, not stored separately

That makes the idempotent representation canonical and keeps redundant state out of the type.

Then:
- add = componentwise
- mul = componentwise
- reciprocal = componentwise reciprocal, if all nonzero

This is the direct generalization of your bicomplex alpha, beta.

Why this is good
- multiplication is trivial
- units / zero divisors are trivial
- ring laws become very easy
- ℂ-vector space structure is immediate
- printing and equality are straightforward in idempotent coordinates

Why this is slightly less romantic
- standard basis is no longer primary
- users lose the immediate feeling of “this is $a + bi + cj + \ldots$”
- you need transforms if you want standard coefficients

Still, for Kosmos, I think this is the better skeleton.

### The real structural point

Your bicomplex type is really a specialized ergonomic facade for the algebra

$$\mathbb{C} \oplus \mathbb{C}$$

A general multicomplex implementation should probably invert the priority:
- core representation: idempotent components
- derived views: recursive pair view, standard coefficient view, named-basis pretty view for small $n$

That would keep the implementation sane.

### A clean abstraction sketch

I would use a core type like this:
- canonical storage in primitive idempotent coordinates
- level inferred from components.size
- recursive view exposed as a derived split, not as the primary representation
- zero(level) and one(level) provided in the companion
- join(left, right) provided as the inverse of split()


```kotlin
@ConsistentCopyVisibility
data class MultiComplex private constructor(
    val components: List<Complex>
) {
    init {
        require(components.isNotEmpty())
        require(components.size.countOneBits() == 1)
    }

    val complexDimension: Int
        get() = components.size

    val level: Int
        get() = 1 + Integer.numberOfTrailingZeros(complexDimension)

    fun idempotent(): List<Complex> =
        components

    fun split(): Pair<MultiComplex, MultiComplex>

    fun isZeroDivisor(): Boolean =
        components.any { it == ComplexAlgebras.ComplexField.zero }

    fun isUnit(): Boolean =
        !isZeroDivisor()

    companion object {
        fun ofIdempotent(components: List<Complex>): MultiComplex =
            MultiComplex(components)

        fun fromComplex(z: Complex): MultiComplex =
            MultiComplex(listOf(z))

        fun join(left: MultiComplex, right: MultiComplex): MultiComplex

        fun zero(level: Int): MultiComplex
        fun one(level: Int): MultiComplex
    }
}
```

Then the ring is just:
```kotlin
override val add = ...
override val mul = ...
override val reciprocal = ...
```

all componentwise.

Then for small levels:
- Complex
- Bicomplex
- maybe Tricomplex

you can provide nice adapters or type aliases or wrappers.

### Where I would be blunt

Here is the blunt version: if you try to generalize the surface form of Bicomplex directly, it will get dumb fast.

Meaning:
- named fields like $a,b,c,d,\ldots$
- hand-written `conj1`, `conj2`, ...
- custom standard-basis conversions per type
- hand-built printable labels
- dedicated matrix monomorphisms for each level

That road becomes a swamp with decorative signage.

But if you generalize the structural idea behind your implementation, then yes, it extends beautifully.

One subtle point: bicomplex is not “just” a recursive complex pair

You can define bicomplex recursively as `Complex + Complex·j`, and you can keep going recursively for multicomplex numbers.

But your current implementation is better than that in one major respect:
- It already exposes the idempotent splitting, which makes invertibility and multiplication much cleaner.

That is why I would not object to multicomplex numbers.
I would object only to building them without making the idempotent decomposition central.

### Recommendation for Kosmos

My opinion:
1. Keep Bicomplex as a polished special case.
2. Introduce a general MultiComplex core type backed by `List<Complex>` in primitive idempotent coordinates.
3. Define:
   - `CommutativeRing<MultiComplex>`
   - `HasReciprocal<MultiComplex>`
   - `Eq<MultiComplex>` and `Printable<MultiComplex>`
   - `FiniteVectorSpace<Complex, MultiComplex>`
   - `FiniteVectorSpace<Real, MultiComplex>`
   - a diagonal `RingMonomorphism<MultiComplex, DenseMat<Complex>>`
4. Treat conjugations generically as actions induced by subsets of generators. 
5. Keep standard-basis coefficient expansion as a derived utility, not as the primary storage model. 
6.	Provide named small-dimensional ergonomics only where they genuinely pay off, especially for Bicomplex.


### Very compact mathematical summary

You can extend it because bicomplex numbers are the second stage of the commutative multicomplex tower, and the idempotent decomposition that makes your bicomplex multiplication so nice generalizes to a decomposition

$$\mathbb{M}_n \cong \mathbb{C}^{2^{n-1}}$$

so the whole structure can be implemented cleanly. The danger is not mathematical failure. The danger is API baroque-ification, where every extra imaginary unit doubles the room count and invites fifteen new conjugations to dinner.

If you want, next I can sketch a concrete `MultiComplex` Kotlin design for Kosmos that stays aligned with your existing `Bicomplex` code and avoids the ornamental swamp.

### The big design choice

Here is the design I would actually recommend for Kosmos, with the sharp edges exposed instead of wrapped in velvet.

I would make general multicomplex numbers use the idempotent $ℂ^m$ representation as the core storage model, where

$$m = 2^{n-1}$$

for level $n$.

Storage invariant: `components.size = 2^(level - 1)` with `level >= 1`.
- Core representation is idempotent and canonical. Standard-basis and recursive views are derived utilities, not alternative storage models.
- Level 1 should agree exactly with Complex, not merely abstractly. Treat this as a compatibility target in tests and API design.

So:
- level 1: Complex ≅ ℂ
- level 2: Bicomplex ≅ ℂ²
- level 3: Tricomplex as a general MultiComplex value in ℂ⁴
- level 4: level-4 multicomplex values in ℂ⁸

This is the cleanest model because:
- addition is componentwise
- multiplication is componentwise
- reciprocal is componentwise
- zero divisors are obvious / explicit
- the decomposition matches the algebraic structure directly

The recursive $x + y j_n$ view should still exist, but only as a derived view, not the storage model.

## 1. Core type

I would start with something like this:

```kotlin
package org.vorpal.kosmos.hypercomplex.multi

import org.vorpal.kosmos.hypercomplex.complex.Complex
import org.vorpal.kosmos.hypercomplex.complex.ComplexAlgebras

@ConsistentCopyVisibility
data class MultiComplex private constructor(
    val components: List<Complex>
) {
    init {
        require(components.isNotEmpty()) { "Multicomplex components must be nonempty." }
        require(components.size.countOneBits() == 1) {
            "The number of idempotent components must be a power of two."
        }
    }

    /**
     * The number of primitive idempotent Complex components.
     * This is 2^(level - 1).
     */
    val complexDimension: Int
        get() = components.size

    /**
     * Level n in the multicomplex tower M_n.
     *
     * level = 1 means ordinary complex numbers
     * level = 2 means bicomplex numbers
     * level = 3 means tricomplex numbers
     * etc.
     */
    val level: Int
        get() = 1 + Integer.numberOfTrailingZeros(complexDimension)

    fun idempotent(): List<Complex> =
        components

    fun isZeroDivisor(): Boolean =
        components.any { ComplexAlgebras.eqComplexStrict(it, ComplexAlgebras.ComplexField.zero) }

    fun isUnit(): Boolean =
        !isZeroDivisor()

    companion object {
        fun ofIdempotent(components: List<Complex>): MultiComplex =
            MultiComplex(components)

        fun fromComplex(z: Complex): MultiComplex =
            MultiComplex(listOf(z))

        fun zero(level: Int): MultiComplex {
            require(level >= 1) { "Multicomplex level must be at least 1." }
            val size = 1 shl (level - 1)
            return MultiComplex(List(size) { ComplexAlgebras.ComplexField.zero })
        }

        fun one(level: Int): MultiComplex {
            require(level >= 1) { "Multicomplex level must be at least 1." }
            val size = 1 shl (level - 1)
            return MultiComplex(List(size) { ComplexAlgebras.ComplexField.one })
        }
    }
}
```

### Why this is good

This type is brutally honest about what multicomplex numbers are algebraically.

It does not pretend that at high levels we still want cute named fields like $a, b, c, d, e, f, g, h$.

That way lies regret and an eel with a clipboard.

## 2. Ring structure

This part is gloriously boring, which is exactly what you want.

```kotlin
package org.vorpal.kosmos.hypercomplex.multi

import org.vorpal.kosmos.algebra.structures.AbelianGroup
import org.vorpal.kosmos.algebra.structures.CommutativeMonoid
import org.vorpal.kosmos.algebra.structures.CommutativeRing
import org.vorpal.kosmos.algebra.structures.HasReciprocal
import org.vorpal.kosmos.core.Symbols
import org.vorpal.kosmos.core.ops.BinOp
import org.vorpal.kosmos.core.ops.Endo
import org.vorpal.kosmos.hypercomplex.complex.ComplexAlgebras
import java.math.BigInteger

object MultiComplexAlgebras {

    fun multicomplexRing(level: Int): CommutativeRing<MultiComplex> =
        MulticomplexCommutativeRing(level)

    class MulticomplexCommutativeRing(
        private val level: Int
    ) : CommutativeRing<MultiComplex>, HasReciprocal<MultiComplex> {

        private val complexField = ComplexAlgebras.ComplexField

        override val zero: MultiComplex = MultiComplex.zero(level)
        override val one: MultiComplex = MultiComplex.one(level)

        private fun requireSameLevel(x: MultiComplex, y: MultiComplex) {
            require(x.level == level) { "Expected level $level but found ${x.level}." }
            require(y.level == level) { "Expected level $level but found ${y.level}." }
        }

        override val add = AbelianGroup.of(
            identity = zero,
            op = BinOp(Symbols.PLUS) { x, y ->
                requireSameLevel(x, y)
                MultiComplex.ofIdempotent(
                    x.components.zip(y.components).map { (a, b) -> complexField.add(a, b) }
                )
            },
            inverse = Endo(Symbols.MINUS) { x ->
                require(x.level == level) { "Expected level $level but found ${x.level}." }
                MultiComplex.ofIdempotent(
                    x.components.map(complexField.add.inverse::invoke)
                )
            }
        )

        override val mul = CommutativeMonoid.of(
            identity = one,
            op = BinOp(Symbols.ASTERISK) { x, y ->
                requireSameLevel(x, y)
                MultiComplex.ofIdempotent(
                    x.components.zip(y.components).map { (a, b) -> complexField.mul(a, b) }
                )
            }
        )

        override fun hasReciprocal(a: MultiComplex): Boolean {
            require(a.level == level) { "Expected level $level but found ${a.level}." }
            return a.isUnit()
        }

        override val reciprocal: Endo<MultiComplex> = Endo(Symbols.INVERSE) { x ->
            require(x.level == level) { "Expected level $level but found ${x.level}." }
            if (!hasReciprocal(x)) {
                throw ArithmeticException("Cannot take reciprocal of a non-invertible multicomplex number.")
            }
            MultiComplex.ofIdempotent(
                x.components.map(complexField.reciprocal::invoke)
            )
        }

        override fun fromBigInt(n: BigInteger): MultiComplex {
            val z = complexField.fromBigInt(n)
            return MultiComplex.ofIdempotent(List(1 shl (level - 1)) { z })
        }
    }
}
```

### Why I like this

It is simple enough that you can trust it.

That matters. Fancy abstractions are nice until they quietly introduce a sign error three layers down and you end up spiritually living inside **Open Wheel.kt, Sebastian.**

## 3. Recursive view as a derived structure

Even though I would not store multicomplex numbers recursively, I would expose the recursive split:

$$M_n \cong M_{n-1} \oplus M_{n-1} j_n$$

because it is mathematically important and useful for printing / human reasoning.

So I would add:

```kotlin
data class MultiComplexPair(
    val x: MultiComplex,
    val y: MultiComplex
)
```

and a method like:
```kotlin
fun MultiComplex.asPair(): MultiComplexPair {
    require(level >= 2) { "Cannot split level-1 Complex view into a multicomplex pair." }
    val half = components.size / 2
    return MultiComplexPair(
        x = MultiComplex.ofIdempotent(components.subList(0, half)),
        y = MultiComplex.ofIdempotent(components.subList(half, components.size))
    )
}
```

### Important note

This pair split is not yet the same thing as the standard-basis form $x + y j_n$ unless you define the corresponding basis convention carefully.

That is one reason I would keep this as a derived view and not pretend it is canonical by default.

## 4. Bicomplex as a special ergonomic wrapper

I would keep your `Bicomplex` type as the polished special case.
- Do not rush Tricomplex as a bespoke wrapper.
- Bicomplex earns its special status because its named units, conjugations, and coordinates are still human-sized.
- A general MultiComplex(level = 3) is enough until there is a real ergonomic reason to add more.

Not because it is strictly necessary, but because it is genuinely useful:
- it has meaningful named units $i, j, k$
- standard coordinates `[a, b, c, d]` are still human-sized
- the three conjugations are important and manageable
- printing in the basis $\{1, i, j, k\}$ is elegant

So I would not delete or replace `Bicomplex`.

Instead I would make it interoperable with `MultiComplex`.

For example:

```kotlin
fun Bicomplex.toMultiComplex(): MultiComplex =
    MultiComplex.ofIdempotent(listOf(alpha, beta))

fun MultiComplex.toBicomplex(): Bicomplex {
    require(level == 2) { "Only level-2 multicomplex values can be converted to Bicomplex." }
    return Bicomplex.ofIdempotent(components[0], components[1])
}
```

That gives you both:
- a polished special-purpose type
- a scalable general-purpose type

That is the right compromise.

## 5. Conjugations
- Do not start implementation here.
- Generic conjugations should come only after the conversion machinery is correct.
- If level-2 conversion is not nailed down first, generic conjugations will become a swamp of sign and ordering bugs.

This is one of the biggest reasons not to hand-roll separate classes forever.

For multicomplex numbers with generators $i1, i2, \ldots,$ every subset of generators gives an involution that flips the sign of those generators.

So conjugations should be represented generically, probably by a bitmask or set of generator indices.

Something like:

```kotlin
data class MulticomplexConjugation(
    val flippedGenerators: Set<Int>
)
```

or, more efficiently:

```kotlin
@JvmInline
value class GeneratorMask(val bits: Int)
```

Then:
```kotlin
fun conjugate(z: MultiComplex, mask: GeneratorMask): MultiComplex
```

### But here is the catch

In the idempotent basis, conjugation is not merely coordinatewise complex conjugation in every case.

Some conjugations:
- permute idempotent coordinates
- some also complex-conjugate entries
- the action depends on how you identify the primitive idempotents

So you need a systematic indexing convention for components.

This is very doable, but it must be designed once, correctly, and then left alone like a temple stone.

I would likely index primitive idempotents by bitstrings of length $n-1$, because that meshes naturally with the decomposition

$$M_n \cong \mathbb{C}^{2^{n-1}}$$

Then generator actions become permutations and conjugations determined by those bits.

### My recommendation

Do not rush the generic conjugation API on day one.

Start with:
- ring structure
- reciprocal
- standard / idempotent conversions
- printables
- compatibility with Complex and Bicomplex

Then add generic involutions once the component indexing convention is fixed and tested.

That is the less glamorous but more intelligent path.

That is excellent for Kosmos.

fun multicomplexComplexVectorSpace(level: Int): FiniteVectorSpace<Complex, MultiComplex>

## 7. Equality and printable

These should also be generic.

### Equality

Approximate and strict equality can just be componentwise using `Eq<Complex>`.

### Printable

There are really two print modes worth supporting:

1. Mode A: idempotent printable

Print as something like:

$$(\alpha_0, \alpha_1, \ldots, \alpha_{m-1})$$

This is ideal for algebra and debugging.

2. Mode B: standard-basis printable

Print as:

$$a_0 + a_1 i_1 + a_2 i_2 + a_3 i_1 i_2 + \cdots$$

This is more intuitive, but requires conversion from idempotent coordinates to standard coefficients.

I would implement Mode A first because it is canonical, easy, and honest.

Later, implement a Walsh-Hadamard-style transform to get standard coefficients.

⸻

## 8. Standard-basis conversion

This is the one place where the gremlins start whispering.
- The conversion should be treated as a recursive complex-twisted Walsh-Hadamard transform.
- The right implementation target is:
	- `standardRealsToIdempotent(level, coeffs)`
	- `idempotentToStandardReals(z)`
- These should be tested as mutual inverses before they are used by printables or conjugations.

For bicomplex numbers, you manually derived:
- standard basis ↔ idempotent basis

For general multicomplex numbers, this transform becomes recursive and sign-patterned.

**The good news:** it is not mysterious. It is just a structured linear transform.

**The bad news:** if you hand-wave it, you will create a beautiful little nest of off-by-sign nonsense.

### What I would do

Define the recursive standard basis:

$$M_n = M_{n-1} \oplus M_{n-1} j_n$$

Then write conversions recursively:
- standard coefficients of level $n$
- idempotent components of level $n$

using the splitting induced by the idempotents of the newest unit.

This is entirely implementable, but I would keep it out of the initial core until the storage model and indexing convention are nailed down.

⸻

## 9. Matrix monomorphisms

For general multicomplex numbers, the clean analogue of your bicomplex diagonal monomorphism is almost embarrassingly easy:

$$M_n \hookrightarrow M_{2^{n-1}}(\mathbb{C})$$

as diagonal matrices.

So:

```kotlin
fun multicomplexToDiagonalMatrixMonomorphism(level: Int): RingMonomorphism<MultiComplex, DenseMat<Complex>>
```

mapping components to a diagonal matrix.

That is clean, canonical, and more useful than trying to produce cute special companion-matrix representations at every level.

The diagonal representation is the right default.

Testing doctrine
•	Do not trust pretty formulas without roundtrip tests.
•	Minimum required checks:
•	level 1 agrees with Complex
•	level 2 agrees with existing Bicomplex
•	standard → idempotent → standard is identity
•	idempotent → standard → idempotent is identity
•	multiplication is preserved under conversion
•	Treat level 2 as the calibration chamber for all later conventions.

## 12. One thing I would not do

I would not make `MultiComplex<A>` generic over the base scalar ring.

That sounds attractive for a hot minute, but it muddies the concept.

Multicomplex numbers are specifically the tower built from commuting square roots of `-1`, and the nice idempotent decomposition lands in ordinary complex numbers. The type should say that plainly.

If later you want a more general “commuting quadratic extension” tower, that is a different abstraction and deserves its own name.

Do not make `MultiComplex` into a template bazooka just because Kotlin technically lets you suffer.

## 14. My real opinion

So, bluntly:

You can extend bicomplex numbers to multicomplex numbers, and you probably should, because the generalization is natural and elegant.

My only objection is this:

Do not generalize the decorative surface of Bicomplex; generalize the algebraic skeleton underneath it.
- Bicomplex is a lovely hand-carved box.

## Part II. The interesting machinery: standard basis ↔ idempotent basis

This is the real fun.

The key fact is that multicomplex numbers are built from commuting imaginary units
$$i_1, i_2, \dots, i_n$$
with each:
$$i_k^2 = -1.$$

But the primitive idempotents arise from the hyperbolic units
$$e_k = i_1 i_k \quad \text{for } k = 2, \dots, n$$
and these satisfy
$$e_k^2 = 1.$$
- Index primitive idempotents by bitstrings of length n - 1, not by ad hoc labels.
- This indexing should drive:
    - component order
    - conjugation permutations later
    - deterministic printing/debugging

Each such hyperbolic unit yields orthogonal idempotents

$$\varepsilon_k^{+} = \frac{1 + e_k}{2}, \qquad
\varepsilon_k^{-} = \frac{1 - e_k}{2}.$$

Then a primitive idempotent is a product:
$$\varepsilon_{\sigma}
=
\prod_{k=2}^n \varepsilon_k^{\sigma_k}$$

where each $$\sigma_k \in \{+,-\}$$.

There are $2^{n-1}$ such primitive idempotents, and they split the algebra as

$$\mathbb{M}_n \cong \bigoplus_{\sigma \in \{\pm\}^{n-1}} \mathbb{C}\,\varepsilon_\sigma
\cong \mathbb{C}^{2^{n-1}}.$$

That is the deep structural reason the idempotent model works.

⸻

## 1. The standard basis

A standard monomial basis over ℝ is indexed by subsets of \{1,\dots,n\}:

$$i_S := \prod_{k \in S} i_k$$

with $i_\varnothing = 1$.

So an arbitrary multicomplex number is

$$z = \sum_{S \subseteq \{1,\dots,n\}} a_S i_S$$

with real coefficients $a_S$. That is a real basis of size $2^n$.

For example, at level 2:

$$z = a + b i_1 + c i_2 + d i_1 i_2$$

which is your bicomplex standard basis.

⸻

## 2. The idempotent basis

For each sign pattern $\sigma \in \{\pm 1\}^{n-1}$, define $\varepsilon_\sigma$ as above.

Then every multicomplex number has a unique idempotent decomposition

$$z = \sum_{\sigma \in \{\pm 1\}^{n-1}} \alpha_\sigma \varepsilon_\sigma$$

where each $\alpha_\sigma \in \mathbb{C}(i_1)$.

These $\alpha_\sigma$ are precisely the components: `List<Complex>`.

So the two coordinate systems are:
- standard real coefficients $a_S$
- idempotent complex coefficients $\alpha_\sigma$

⸻

## 3. The explicit transform

This is the juicy part.

Take a standard-basis element
$$i_S$$
and ask how it acts on the idempotent component indexed by $\sigma$.

Write $S$ either as:
- not containing 1, or
- containing 1.

Then on the idempotent $\varepsilon_\sigma$, we have:

$$i_k \varepsilon_\sigma = \sigma_k\, i_1 \varepsilon_\sigma
\qquad (k \ge 2)$$

where $\sigma_k \in \{\pm 1\}$ is the sign in the idempotent choice.

Therefore any monomial $i_S$ collapses on the $\sigma$-component to either:
- a real sign, or
- that sign times $i_1$.

Concretely, if $S \subseteq \{1,\dots,n\}$, let:
- $T = S \setminus \{1\}$
- $m = |T|$

Then on the $\sigma$-component,

$$i_S \mapsto \left(\prod_{k \in T} \sigma_k \right) i_1^{\,\mathbf{1}_{1 \in S} + m}$$

Since $i_1^2 = -1$, this is always one of:

$$\pm 1, \pm i_1.$$

So each idempotent component is a complex linear combination of the real standard coefficients.

This is basically a complex-valued Walsh-Hadamard transform.

⸻

## 4. The forward formula

Suppose
$$z = \sum_{S \subseteq \{1,\dots,n\}} a_S i_S.$$

Then the idempotent component $\alpha_\sigma$ is:

$$\alpha_\sigma = \sum_{S \subseteq \{1,\dots,n\}} a_S \, \chi_\sigma(S)$$

where $\chi_\sigma(S) \in \{ \pm 1, \pm i\}$ is the evaluation of $i_S$ under the sign pattern $\sigma$.

More explicitly:

$$\chi_\sigma(S) = \left(\prod_{k \in S \setminus \{1\}} \sigma_k \right) i^{\,|S \cap \{1\}| + |S \setminus \{1\}|}.$$

This formula is correct in substance, though for implementation I would use a recursive transform rather than compute it from scratch every time.

⸻

## 5. The bicomplex case drops out perfectly

For $n=2$, there is one sign $\sigma_2 \in \{\pm 1\}$, so two idempotents.

Write:

$$z = a + b i_1 + c i_2 + d i_1 i_2.$$

Then:
- on the $+$ component, $i_2 = + i_1$
- on the $-$ component, $i_2 = - i_1$

and $i_1 i_2$ becomes:
$$+: i_1^2 = -1$$
$$-: -i_1^2 = +1$$

So the two complex components are:

$$\alpha_+ = (a - d) + (b + c)i$$
$$\alpha_- = (a + d) + (b - c)i$$

That is exactly your bicomplex formula, just written with a slightly different indexing convention.

Your current code has:

$$\alpha = (a - d) + (b + c)i$$
$$\beta = (a + d) + (b - c)i$$

Beautiful. The machinery is already there. It was quietly wearing a fake moustache.

- The level-2 generic conversion must be checked against the existing formulas:
$$\alpha = (a - d) + (b + c)i$$
$$\beta = (a + d) + (b - c)i$$
- If the generic convention yields the same result up to component swap, either:
	- change the convention so that it matches exactly, or
	- document the bridge explicitly and test it.

⸻

## 6. The recursive transform is the practical way

The best implementation is recursive, not formula-sledgehammered.

Suppose a level $n$ multicomplex number is written recursively as

$$z = x + y i_n$$

where $x, y \in M_{n-1}$.

Now form the hyperbolic unit

$$e_n = i_1 i_n.$$

Its idempotents split according to the sign of $e_n$. On the $+$ branch, $i_n = -i_1$ or $+i_1$ depending on convention; on the other branch, the opposite sign appears.

You then get two recursive components:

$$\alpha_+ = x + y \cdot \lambda_+$$
$$\alpha_- = x + y \cdot \lambda_-$$

where $\lambda_\pm = \pm i_1$ in the $level-n-1$ algebra.

If you choose conventions consistently, the recursive rule becomes something like:
$$
\mathrm{idemp}(x + y i_n)
=
\bigl(
\mathrm{idemp}(x - i_1 y),
\mathrm{idemp}(x + i_1 y)
\bigr)
$$
or the sign-flipped variant depending on basis ordering.

This is the multicomplex analogue of the 2-point Hadamard butterfly.

So every level doubles the number of components by applying:
- one branch with $x - i*y$
- one branch with $x + i*y$

recursively.

That is gorgeous and efficient.

⸻

## 7. Recursive algorithm sketch

Let standard mean recursive pair form over the newest unit.

So level $n$ is represented as:

```kotlin
data class StandardMultiComplex(
    val left: StandardMultiComplex?,
    val right: StandardMultiComplex?,
    val complex: Complex?
)
```

or more sanely by raw real coefficient lists.

Then the recursive transform to idempotent coordinates looks like:

```kotlin
fun standardToIdempotent(level: Int, coeffs: List<Complex>): List<Complex> {
    if (level == 1) {
        return coeffs
    }

    val half = coeffs.size / 2
    val x = standardToIdempotent(level - 1, coeffs.subList(0, half))
    val y = standardToIdempotent(level - 1, coeffs.subList(half, coeffs.size))

    return x.zip(y).map { (a, b) -> a - i * b } +
        x.zip(y).map { (a, b) -> a + i * b }
}
```

Conceptually, that is the transform.

But to match your preferred standard real basis exactly, we need to be careful about what the recursive coeffs mean. The cleanest practical route is:
- represent standard basis as real coefficients indexed by bitmasks
- recursively split by whether the monomial includes the newest unit
- bundle real coefficients into lower-level multicomplex pieces
- then apply the butterfly

⸻

## 8. Inverse transform

The inverse is just the inverse butterfly.

If:

$$\alpha_- = x - i y,\qquad \alpha_+ = x + i y$$

then:

$$x = \frac{\alpha_- + \alpha_+}{2}$$
$$y = \frac{\alpha_+ - \alpha_-}{2i}$$

Since
$$\frac{1}{i} = -i,$$
this is

$$y = -\frac{i}{2}(\alpha_+ - \alpha_-).$$

That gives the recursive inverse transform from idempotent components back to standard recursive coordinates.

Again: butterfly in, butterfly out. Tiny algebraic origami.

⸻

## 9. A concrete implementation sketch of the conversion utilities

Here is a realistic utility object.

MultiComplexConversions.kt
```kotlin
package org.vorpal.kosmos.hypercomplex.multi

import org.vorpal.kosmos.core.math.Real
import org.vorpal.kosmos.hypercomplex.complex.Complex
import org.vorpal.kosmos.hypercomplex.complex.ComplexAlgebras
import org.vorpal.kosmos.hypercomplex.complex.complex

object MultiComplexConversions {

    private val complexField = ComplexAlgebras.ComplexField

    /**
     * Labels for the standard real basis:
     *
     * level 1: ["", "i1"]
     * level 2: ["", "i1", "i2", "i1i2"]
     * level 3: ["", "i1", "i2", "i1i2", "i3", "i1i3", "i2i3", "i1i2i3"]
     *
     * Ordered by subset bitmask from 0 to 2^level - 1.
     */
    fun standardBasisLabels(level: Int): List<String> =
        (0 until (1 shl level)).map { mask ->
            if (mask == 0) ""
            else (0 until level)
                .filter { bit -> ((mask shr bit) and 1) == 1 }
                .joinToString("") { bit -> "i${bit + 1}" }
        }

    /**
     * Convert standard real coefficients (indexed by subset bitmask)
     * into idempotent Complex coefficients.
     *
     * Input size must be 2^level.
     * Output size is 2^(level - 1).
     */
    fun standardRealsToIdempotent(level: Int, coeffs: List<Real>): MultiComplex {
        require(level >= 1) { "Level must be at least 1." }
        require(coeffs.size == (1 shl level)) {
            "Expected ${1 shl level} real coefficients for level $level but found ${coeffs.size}."
        }

        return if (level == 1) {
            MultiComplex.fromComplex(complex(coeffs[0], coeffs[1]))
        } else {
            val half = coeffs.size / 2

            val x = standardRealsToIdempotent(level - 1, coeffs.subList(0, half))
            val y = standardRealsToIdempotent(level - 1, coeffs.subList(half, coeffs.size))

            val i = ComplexAlgebras.ComplexField.i

            val left = x.components.zip(y.components).map { (a, b) ->
                complexField.add(a, complexField.add.inverse(complexField.mul(i, b)))
            }

            val right = x.components.zip(y.components).map { (a, b) ->
                complexField.add(a, complexField.mul(i, b))
            }

            MultiComplex.ofIdempotent(left + right)
        }
    }

    /**
     * Convert idempotent Complex coefficients back into standard real coefficients.
     */
    fun idempotentToStandardReals(z: MultiComplex): List<Real> =
        idempotentToStandardRealsRecursive(z.components)

    private fun idempotentToStandardRealsRecursive(components: List<Complex>): List<Real> {
        return if (components.size == 1) {
            listOf(components[0].re, components[0].im)
        } else {
            val half = components.size / 2
            val left = components.subList(0, half)
            val right = components.subList(half, components.size)

            val x = left.zip(right).map { (aMinus, aPlus) ->
                complex(
                    (aMinus.re + aPlus.re) / 2,
                    (aMinus.im + aPlus.im) / 2
                )
            }

            val y = left.zip(right).map { (aMinus, aPlus) ->
                // y = (aPlus - aMinus) / (2i) = -(i/2)(aPlus - aMinus)
                val diff = complex(
                    aPlus.re - aMinus.re,
                    aPlus.im - aMinus.im
                )
                complex(
                    diff.im / 2,
                    -diff.re / 2
                )
            }

            idempotentToStandardRealsRecursive(x) + idempotentToStandardRealsRecursive(y)
        }
    }
}
```

## 10. Important convention warning

Here is where the dragon actually lives.

The recursion above assumes a very specific convention for:
- ordering standard basis monomials
- splitting by the newest generator
- branch order in idempotent coordinates
- whether the butterfly uses $x - i y / x + i y$ or the opposite order

## Multicomplex implementation plan for Kosmos

The guiding rule:

- Make the idempotent core correct first.
- Everything fancy is downstream of that.

## 1. Generator naming
Use:
$$\{i_1, i_2, ..., i_n\}$$
- all commute
- each squares to $-1$

## 2. Standard basis indexing

Index standard real coefficients by bitmask over:
$$\{i_1, ..., i_n\}.$$

For example at level 3:
- `0b000 -> 1`
- `0b001 -> i1`
- `0b010 -> i2`
- `0b011 -> i1i2`
- `0b100 -> i3`
- `0b101 -> i1i3`
- `0b110 -> i2i3`
- `0b111 -> i1i2i3`

That is clean and machine-friendly.

## 3. Idempotent indexing

Primitive idempotents are indexed by sign choices for the hyperbolic units:

$$e_k = i_1 i_k \quad (k = 2,\dots,n)$$

So there are $$2^{n-1}$$ idempotent components, indexed by bitmasks of length $$n - 1.$$

## 4. Recursive butterfly convention

Choose exactly one convention for standard → idempotent.

For example:

$$(x, y) \mapsto (x - i y,\; x + i y)$$

where $i$ means the ordinary complex unit in the idempotent coordinates.

This must be fixed once and then obeyed everywhere.

## 5. Bicomplex compatibility target

Decide whether the generic level-2 conversion must match your current Bicomplex($\alpha, \beta$):
- exactly, or
- up to swapping the two idempotent components

My recommendation: make it match exactly if practical. If not, document the conversion bridge explicitly.

This document is small, but it is the fence around the garden.
